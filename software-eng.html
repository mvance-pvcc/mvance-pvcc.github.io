Software Engineering
Software Engineering The branch of computer science that seeks principles to guide the development of large, complex software systems.
Integrated Development Environment (IDE) Combined tools for developing software (editors, compilers, debugging tools, and so on) into a single, integrated package.
Association for Computing Machinery (ACM) Purpose: Dedicated to advancing the arts, sciences, and applications of information technology.
Benefits: Networking opportunities, ACM's Career & Job Center, ability to join various ACM special interest groups, and much more.
Student ACM Membership: Student membership is as simple as filling out a form and paying a subscription fee!
Institute of Electrical and Electronis Engineers (IEEE) Purpose: To foster technological innovation and excellence for the benefit of humanity.
Benefits: Networking, build a professional profile, discuss various technical interests, and much more!
Student IEEE Membership: A Student/Graduate Student member must carry at least 50% of a normal full-time academic program as a registered undergraduate or graduate student in a regular course of study in IEEE designated fields.
Software Life Cycle: Development Phase Requirements Analysis: To specify what services the proposed system will provide, to identify any conditions (time constraints, security, and so on) on those services, and to define how the outside world will interact with the system.
Design: Creating a plan for the construction of the proposed system.
Implementation: Writing of programs, creation of data files, and development of databases.
Testing: The process of debugging programs and confirming that the final software product was compatible with the software requirements specification.

Unified Modeling Language (UML) A general-purpose, developmental modeling language in the field of software engineering that is intended to provide a standard way to visualize the design of a system
Use Case Interactions within a use case diagram.
Use Case Diagram A use case diagram depicts the proposed system as a large rectangle in which interactions (called use cases) between the system and its users are represented as ovals and users of the system (called actors) are represented as stick figures (even though an actor may not be a person)
Data Relationships One-to-one: A one-to-one relationship is exemplified by the association between patients and occupied private rooms in that each patient is associated with only one room and each room is associated with only one patient.
One-to-many: A one-to-many relationship is exemplified by the association between physicians and patients in that one physician is associated with many patients and each patient is associated with one (primary) physician.
Many-to-many: A many-to-many relationship would occur if we included consulting physicians in the physician-patient relationship. Then each physician could be associated with many patients and each patient could be associated with many physicians.
Design Pattern A design pattern is a predeveloped model for solving a recurring problem in software design.
Software Quality Assurance (SQA) Charged with overseeing and enforcing the quality control systems adopted by the organization.
Software Testing Glas Box Testing: The software tester is aware of the interior structure of the software and uses this knowledge when designing the test.
Black Box Testing: Tests that do not rely on knowledge of the software’s interior composition.
Beta Testing: A preliminary version of the software is given to a segment of the intended audience with the goal of learning how the software performs in real-life situations before the final version of the product is solidified and released to the market.
Alpha Testing: Similar testing performed at the developer’s site is called alpha testing.
Documentation User Documentation: The purpose of user documentation is to explain the features of the software and describe how to use them.
System Documentation: The purpose of system documentation is to describe the software’s internal composition so that the software can be maintained later in its life cycle.
Technical Documentation: The purpose of technical documentation is to describe how a software system should be installed and serviced.
Intellectual Property Law Software License: A software license is a legal agreement between the owner and user of a software product that grants the user certain permissions to use the product without transferring ownership rights to the intellectual property.
Software Disclaimer: Software developers often include disclaimers in the software licenses that state the limitations of their liability.
Best Defenses Against Sofware Liability Claims: Apply sound software engineering principles during the software’s development, to use a level of care compatible with the software’s application, and to produce and maintain records that validate these endeavors.
